## 零、v1 审计中暴露的缺陷

v1 审计完成了 5 层 25 项检查 + 补充审计，但在后续验证中仍发现了严重遗漏：

| 遗漏的 bug | 发现时机 | v1 为何漏掉 |
|-----------|---------|------------|
| **pop_region 文件冲突**：macOS 大小写不敏感导致 `pop_ssp126.xlsx`（区域）被 `pop_SSP126.xlsx`（国家）覆盖 | 测试运行 DSCALE 时 KeyError | 审计在文件移动后未立即重跑验证；未检查路径变更后的实际文件内容 |
| **Other 聚合名误映射**：`_build_iea_name_index` 将 "Other non-OECD Americas" 映射到单个国家（vct 独吞 166,577 TJ）| 用户质疑 "IEA 似乎只有 150 国" | 审计仅检查了匹配数量（B1），未检查匹配质量——即每个名称映射到的 ISO 是否正确 |
| **Kaya 零值塌缩**：30 个无 IEA 基线的国家 2100 年 TFC→0 | 图表空白触发的后续调查 | D6 检查了 "CAGR < -10%" 但未检查 "有 GDP 但 TFC=0" 的逻辑不一致 |
| **无测试套件** | Gap 8 才补充 | 审计先于测试——应该测试先于审计 |
| **虚假 Guinea 匹配**：诊断代码产生假阳性 | 用户质疑 | 审计工具和诊断代码本身未经校验 |

**根因**：v1 是**检查清单驱动**的——逐项打勾但缺乏系统性验证。v2 改为**不变量驱动**——定义代码必须满足的不变条件，通过自动化测试持续验证。

## 一、核心原则

### 1. 测试先于审计

任何审计步骤之前，必须先有一组通过的不变量测试。审计的目标是**扩展**测试覆盖，而非**替代**测试。

```
错误流程（v1）：审计 → 发现问题 → 修复 → 写测试
正确流程（v2）：写测试 → 审计 → 发现遗漏 → 补充测试 → 修复 → 全部重跑
```

### 2. 每次修改后立即端到端验证

任何文件移动、路径变更、配置修改后，必须立即运行完整 pipeline + 测试套件。不允许"先改完再一起测"。

### 3. 双向验证

不仅验证"应该匹配的确实匹配了"，还必须验证"不应该匹配的确实没匹配"。
例如：B1 不仅需要统计匹配率，还需要逐一验证每个映射的 ISO 是否正确。

### 4. 数据流逐点追踪

对每个关键数据点（如特定国家的 TFC 值），从原始文件逐层追踪到最终输出，验证每一步的变换正确。

### 5. 平台差异意识

文件系统（大小写敏感/不敏感）、Python 版本、pandas 版本的行为差异必须在审计中显式考虑。

## 二、不变量层级

审计围绕以下不变量组织，按优先级排列。每个不变量必须有对应的自动化测试。

### I 级 — 物理/数学不变量（违反即致命）

| # | 不变量 | 测试方法 |
|---|--------|---------|
| I1 | Σ_country_in_region(TFC) = GCAM_region(TFC)，∀ 区域, 年份, 情景, 方法 | 参数化测试 |
| I2 | Σ_all_countries(TFC) = Σ_all_regions(TFC)，机器精度内 | 全局求和对比 |
| I3 | TFC ≥ 0，∀ 国家（oth 行除外）| 负值扫描 |
| I4 | 份额 ∈ [0,1]，∀ 份额指标（Logit 方法；Kaya/DSCALE 允许少量偏离）| 边界检查 |

### II 级 — 逻辑一致性不变量（违反表明代码缺陷）

| # | 不变量 | 测试方法 |
|---|--------|---------|
| II1 | 单国区域三方案结果完全一致（≤1e-10 偏差）| 15 个单国区域全量对比 |
| II2 | 有 IEA 基线（>100 TJ）的国家 2100 年 TFC > 0 | 塌缩检测 |
| II3 | 全局 TFC 排序：SSP245 > SSP434, SSP245 > SSP126 | 情景排序验证 |
| II4 | 每个 GCAM 区域至少有一个国家在输出中 | 区域覆盖检查 |
| II5 | 同一 ISO 在同一方法的不同情景间，2015 年值不变 | 基年一致性 |

### III 级 — 数据完整性不变量（违反表明数据或匹配问题）

| # | 不变量 | 测试方法 |
|---|--------|---------|
| III1 | WORLDBAL 中每个国家名最多匹配一个 ISO（唯一映射）| 名称→ISO 反向检查 |
| III2 | 每个 ISO 最多匹配一个 WORLDBAL 名称（无重复）| ISO→名称正向检查 |
| III3 | "Other..." 聚合名不映射到任何单国 | 聚合名泄漏检查 |
| III4 | 每个排除实体有明确排除理由（非主权/数据缺失）| EXCLUDED_ISO 合理性 |
| III5 | 所有 config.py 路径指向存在的文件 | 路径存在性测试 |

### IV 级 — 代码质量不变量

| # | 不变量 | 测试方法 |
|---|--------|---------|
| IV1 | 无死导入 | static analysis |
| IV2 | 无重复函数定义 | AST 对比 |
| IV3 | np.seterr 不泄漏到全局 | 导入前后对比 |
| IV4 | 空输入不崩溃 | 边界测试 |

### V 级 — 跨方法一致性不变量

| # | 不变量 | 测试方法 |
|---|--------|---------|
| V1 | Logit-Kaya log-log r > 0.85（∀ 指标）| 相关性阈值 |
| V2 | 三方案的国家数量一致（排除 oth 行）| ISO 集对比 |
| V3 | 基年（2015）三方案的国家值高度相关（r > 0.99）| 基年一致性 |

## 三、审计流程

### Phase 0：基础设施（必须先于任何审计）

```
0.1 确认测试套件存在并通过（≥N 项不变量测试）
0.2 确认 CI/CD 或手动运行脚本可用
0.3 记录当前 git commit → 所有输出文件 → 文件哈希快照
```

### Phase 1：数据链路审计（基于 III 级不变量）

```
1.1 加载映射文件 → 验证 180 行，32 区域
1.2 遍历每个映射行：
    a. IEA_ctry 是否以 "Other" 开头？→ 标记为聚合国家
    b. IEA_ctry 非 Other？→ 在 WORLDBAL 中查找，验证精确匹配（非子串）
    c. GCAM Country 是否在 WORLDBAL 中？→ 同上
1.3 反向检查：WORLDBAL 中每个 Country/Region → normalize → 查找映射
    a. 找到映射？→ 验证 ISO 正确（抽查 20 个）
    b. 未找到？→ 分类：排除实体 / 历史实体 / 真正未匹配
1.4 对每个 "Other..." 聚合国家：
    a. 验证其基线值来自人口拆分（非 0 且非整个聚合值）
    b. 抽样 vct/bhs/blz 验证值在合理范围（< 聚合总量 / 成员国数 × 3）
1.5 平台检查：确认所有文件名在任何目标平台上唯一（大小写敏感性）
```

### Phase 2：计算链路审计（基于 I/II 级不变量）

```
2.1 对每个指标×方法×情景运行端到端 pipeline
2.2 验证全部 I 级不变量（守恒、非负、有界）
2.3 验证全部 II 级不变量（单国一致、塌缩检测、情景排序）
2.4 数据点追踪：抽样 5 个国家（USA, China, Nigeria, Germany, bfa）
    → 从原始 GCAM + IEA → 中间计算 → 最终输出，逐层验证
2.5 边界测试：空区域、零 GDP、零人口、极端 gamma
```

### Phase 3：跨方法对比审计（基于 V 级不变量）

```
3.1 验证全部 V 级不变量
3.2 生成散点图 + 偏差直方图
3.3 标记方法间分歧 > 2σ 的国家，逐个核查原因
3.4 验证分歧原因可解释（GDP 增长差异 / 区域结构 / 数据缺失）
```

### Phase 4：文档与可复现性

```
4.1 更新 CLAUDE.md 反映当前代码结构
4.2 记录所有已知局限及决策理由
4.3 生成文件哈希清单（输入 + 输出）
4.4 Git tag 审计完成点
```

## 五、测试套件最低要求

审计通过的前提是以下测试全部通过：

```
compare/tests/
├── test_invariants_I.py    # I1-I4: 守恒、非负、有界（~100 项参数化）
├── test_invariants_II.py   # II1-II5: 逻辑一致性（~30 项）
├── test_invariants_III.py  # III1-III5: 数据完整性（~20 项）
├── test_api.py              # 函数签名、返回值结构、导入
├── test_dscale_boundary.py  # fun_max_tc bounds, beta clip, alpha harm
└── test_edge_cases.py       # 空输入、零值、极端 gamma、单国区域
```

最低通过标准：**全部测试通过，0 失败，允许的警告数 ≤3**。

## 七、与 v1 的关键差异

| 维度 | v1 | v2 |
|------|----|----|
| 驱动方式 | 检查清单 | 不变量 + 测试 |
| 测试时机 | 审计后 | 审计前 |
| 验证方式 | 单向（匹配率）| 双向（正确匹配 + 无反匹配）|
| 修改后验证 | 批量修改后 | 每次修改后立即 |
| 平台考虑 | 无 | 显式检查 |
| 数据点追踪 | 无 | 抽样 5 国全链路 |
| 边界测试 | 无 | 空输入/零值/极端参数 |
| 完成标准 | 定性 | 定量（测试数+文件哈希+pipeline OK）|
